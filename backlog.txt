Problem:
  kalshi_client.py's live order-placement and order-cancellation code
  targets endpoints Kalshi is migrating off of. The split is "reads stay,
  writes move," not a wholesale change:
    - GET /portfolio/orders (kalshi_client.py:313, list resting/executed
      orders) and GET /portfolio/orders/{order_id} (kalshi_client.py:423,
      get single order) are UNAFFECTED -- reads stay on this path
      permanently per Kalshi's own docs.
    - POST /portfolio/orders (kalshi_client.py:364, place_order) and
      DELETE /portfolio/orders/{order_id} (kalshi_client.py:427,
      cancel_order) are the legacy WRITE endpoints being deprecated. The
      new V2 write endpoints are POST /portfolio/events/orders and
      DELETE /portfolio/events/orders/{order_id} (confirmed via both
      Kalshi's changelog and an independent third-party SDK
      -- github.com/TexasCoding/kalshi-python-sdk -- whose docs describe
      the identical old-path/new-path split).
  This is NOT just a URL rename: Kalshi's changelog describes the new V2
  write endpoints as using "a bid/ask single-book shape with fixed-point
  dollar prices and lightweight responses" -- a different request/response
  shape. The third-party SDK's docs show the new response includes
  order_id, remaining_count, fill_count, reduced_by, ts_ms -- diff this
  against what kalshi_client.py currently parses out of the old response
  before assuming the migration is small.
  Timeline: legacy write rate-limit costs escalated to 10x V2-equivalent by
  2026-06-04; deprecation was announced for "sometime between June 18 and
  June 25, 2026." No changelog entry confirms the deprecation was actually
  enforced/completed -- legacy writes may still work today, just at 10x
  rate-limit cost, with no confirmed grace period remaining. No real order
  has ever been placed by this bot yet (ENABLE_MICRO_LIVE=False), so this
  has not caused a live incident.

What the fix looks like:
  1. Quick check first: hit place_order/cancel_order against DEMO_BASE
     (https://demo-api.kalshi.co/trade-api/v2) and see whether they still
     succeed, return a deprecation error, or already 404/reject.
  2. Migrate kalshi_client.py's place_order (line 364) and cancel_order

  3. Leave the two GET /portfolio/orders* call sites (line 313, 423) alone

  4. While in there, evaluate the new orderbook_delta WebSocket

Why not now (only reason to defer, not to skip):

[HOURLY-DIRECTIONAL TEMPERATURE MARKETS — KXTEMPxxxH]
Priority: Low/Medium (one city has real live volume right now; the rest don't)

Problem:
  Kalshi's "Climate and Weather" category also lists an "Hourly Directional
  [City] Temperature" market family, structurally invisible to this bot:
  check_series_drift() only ever diffs KXHIGH*/KXLOW* prefixed series, so
  this family never surfaces as a gap no matter how much volume it gets.
  Confirmed live 2026-07-08: KXTEMPMIAH, KXTEMPBOSH, KXTEMPLAXH, KXTEMPCHIH,
  KXTEMPAUSH, KXTEMPDCH, KXTEMPNYCH all exist; only KXTEMPNYCH currently has
  open markets (24 of them, e.g. KXTEMPNYCH-26JUL0804-T76.99 -- an hourly
  threshold, not a daily high/low). The other 6 cities are dormant right now.

What the fix looks like:
  This isn't a ticker-list fix -- it needs a per-hour threshold-crossing
  probability model instead of the daily high/low ensemble blend this bot
  runs today. The good news: the same-day nowcasting infra this would lean
  on (live METAR observation override, persistence baseline, HRRR same-day
  fetch -- see analyze_trade()'s days_out==0 path in weather_markets.py, and
  cron.py's "Same-day markets... re-enabled" same-day trading logic) already
  exists and is live for daily markets, not still-deferred as an earlier
  version of this backlog entry assumed. Still needs new work though: the
  existing pipeline reasons about "will today's high/low cross X," not
  "will the temp at hour H cross X" -- the per-hour target and the shorter
  settlement window are a real design difference, not just reused as-is.

  Calibration follows the same precedent already established for same-day
  vs multi-day: ml_bias.train_all_temperature_scaling() fits a dedicated
  'sameday' T from days_out=0 rows only, with apply_temperature_scaling()
  enforcing zero fallback to the global/multi-day T (their probability
  distributions are too different to share a scale -- see commit chain
  c773474 -> 9f5bb40, 2026-06-02/05). An hour-level model needs the same
  treatment: its own training pool and its own T, not folded into either
  the daily same-day pool or the multi-day pool. Don't assume the existing
  'sameday' calibration path could just absorb hourly rows -- it can't
  without the same care taken to keep the pools separate.

Why not now:
  - No existing model for the hour-level target specifically (only daily).
  - Only 1 of 7 cities in this family is currently even active.
  - Worth confirming KXTEMPxxxH volume is durable (not a one-off) before
    investing in an hour-level model.

When to revisit:
  - If KXTEMPxxxH activity broadens beyond just NYC, or NYC volume proves
    durable over a few weeks.
  - Given the reusable infra already exists, this is lower-effort than it
    looked when first logged -- re-scope before deprioritizing further.

  - check_series_drift()'s KXHIGH*/KXLOW*-only filter is a deliberate

    tickers") -- start with rain given current live volume.

[TRACK ECMWF FORECAST ACCURACY]
Priority: Low (ECMWF already gets a reasonable static/seasonal weight; this is
  about making that weight data-driven like ICON/GFS already are)

Problem:
  ECMWF's per-model forecast accuracy is never captured anywhere, so it can
  never earn a learned weight the way icon_seamless/gfs_seamless can.
  paper.py's _score_ensemble_members() only ever logs icon_forecast_mean and
  gfs_forecast_mean to tracker.ensemble_member_scores (see the comment left
  in model_means there) -- no ecmwf_forecast_mean field exists anywhere in
  the trade-entry -> settlement -> scoring pipeline. _forecast_model_weights()
  and _model_weights() (weather_markets.py) both correctly fall back to a
  static seasonal default for ECMWF forever, since the tracker data that
  would let them do better structurally can't exist yet.

  Note there are TWO separate ECMWF products in play, tracked independently:
  ecmwf_ifs025 (deterministic, used by the daily 3-model blend in
  get_weather_forecast()) and ecmwf_aifs025_ensemble (ML ensemble, used by
  get_ensemble_temps()'s probability blend). Fixing one does NOT give the
  other a learned weight -- they're unrelated model runs on different
  Open-Meteo endpoints (FORECAST_BASE vs ENSEMBLE_BASE).

What the fix looks like:
  1. Decide which ECMWF product to instrument first. Recommend ecmwf_ifs025
     (the daily-blend one) since that's the one that was actually silently
     dropped from fetches until the 2026-07-07 fix (commit 5a3d80b) -- it's
     the more consequential of the two for real trade decisions.
  2. Add an ecmwf_forecast_mean field and thread it through the same path
     icon_forecast_mean/gfs_forecast_mean already take: weather_markets.py's
     analyze_trade() (near _get_consensus_probs(), ~line 4478) -> the analysis
     dict returned to callers -> order_executor.py/main.py's calls into
     paper.open_paper_trade(...) -> paper.py's trade schema -> finally added
     to _score_ensemble_members()'s model_means dict (~paper.py:1153 as of
     2026-07-10).
  3. _get_consensus_probs() (weather_markets.py ~line 4274 as of
     2026-07-10) currently only
     calls its generic, model-agnostic _model_prob_and_mean() helper for
     "icon_seamless" and "gfs_seamless". Extending it for
     ecmwf_aifs025_ensemble would be mechanically easy (same ENSEMBLE_BASE/
     _ensemble_cache infrastructure already used there). Extending it for
     ecmwf_ifs025 instead requires NEW plumbing mirroring this pattern
     against FORECAST_BASE, since that's a different endpoint entirely --
     don't assume this is a one-line change once ecmwf_ifs025 is picked.
  4. Redesign the currently-binary model_consensus check
     (abs(icon_p - gfs_p) > 0.12) for 3 models -- decide between an
     all-pairwise-agreement check or a max-min-spread check. This is a real
     design decision, not just plumbing -- don't rush it.
  5. Once the field exists, add "ecmwf_ifs025": trade.get("ecmwf_forecast_mean")
     to _score_ensemble_members()'s model_means dict.

Why not now:
  - The urgent bug (ECMWF silently excluded from live fetches) is already
    fixed and does not depend on this -- ECMWF now always gets fetched with
    a reasonable seasonal-default weight regardless.
  - This is new instrumentation across ~4 files, not a bug fix -- meaningfully
    more invasive than the fetch-list fix.
  - Won't pay off for weeks even after shipping: needs 10+ fresh settled
    trades per city (same MIN_OBSERVATIONS/min_n gates icon/gfs already use)
    before the learned weight would ever activate.
  - Requires a real design decision (which ECMWF product, how to redesign the
    3-way consensus check) that deserves its own scoping pass, not a
    rushed addition to an unrelated bug-fix session.

When to revisit:
  - Have bandwidth for a multi-file instrumentation change, not a quick fix.
  - Decide up front whether ecmwf_ifs025 (daily blend, drives the probability
    calc real trades actually use) or ecmwf_aifs025_ensemble (ensemble
    consensus data) matters more before starting -- see note above, they are
    not interchangeable.

[CLOSED 2026-07-12 -- evaluated live, not switching] OPEN-METEO ENSEMBLE MEAN
  API -- SIMPLER/CHEAPER THAN CURRENT MEMBER-BY-MEMBER FETCH
Priority: was Medium (efficiency + data-quality improvement, not a bug fix)

CLOSED 2026-07-12: made live test calls to the actual endpoint (not just docs)
  and found three real blockers, not one:
  1. Daily spread does not exist as a variable at all -- requesting
     temperature_2m_max_spread (or any daily _spread variant) 400s with
     "Cannot initialize ForecastVariableDaily from invalid String value."
     Spread is HOURLY-ONLY. Since this bot needs daily-high/low uncertainty,
     using this API would mean fetching hourly spread and picking the value
     near the forecast daily max/min hour -- reintroducing complexity, not
     eliminating it. Not the "get uncertainty for free" win it looked like.
  2. Spread comes back in Kelvin-degrees (hourly_units reports
     "temperature_2m_spread":"K") even when temperature_unit=fahrenheit is
     explicitly requested -- the mean value correctly converts to F, spread
     does not. A Kelvin-degree spread is NOT numerically equal to a
     Fahrenheit-degree spread; using it unconverted as sigma in
     gaussian_probability() would silently understate uncertainty by ~44%
     (missing x1.8 factor). Easy to get wrong silently.
  3. ECMWF AIFS mean (ecmwf_aifs025_ensemble_mean) returned null for daily
     max/min at the one live test point (NYC) -- same failure mode already
     documented in this repo's fetch_temperature_ecmwf() comments for the
     deterministic endpoint.
  Also re-confirmed and corrected the model-name guess yet again on live
  verification: ICON mean is dwd_icon_eps_ensemble_mean_seamless, GFS mean is
  ncep_gefs_ensemble_mean_seamless (NOT gfs_ensemble_seamless_mean -- GFS's
  ensemble is branded GEFS at Open-Meteo, guessing wrong 400s), ECMWF AIFS
  mean is ecmwf_aifs025_ensemble_mean (NOT ecmwf_aifs_025_ensemble_mean).
  Verdict: not a clean drop-in win as originally framed. Closing rather than

[RESTING EXIT ORDERS + OCO ORDER GROUPS]
Priority: High (live risk-management hole, not just an optimization)

Problem:
  All exits are poll-based: check_exit_targets()/check_model_exits()/
  stop-losses only run when a cron or watch cycle happens to execute. On a
  manual-cron cadence (e.g. ~3 runs/day during the Belgium trip) an open
  position can sit through its stop level for hours with zero protection.
  Kalshi's API supports order_groups (OCO — linked orders where one fill
  auto-cancels the others) and a WebSocket `fill` channel; the bot uses
  neither (kalshi_ws.py:301 subscribes only to ticker + orderbook_delta,
  fills are discovered by polling _poll_pending_orders()).

UPDATE 2026-07-12 (research pass, no code changed) -- IMPORTANT CORRECTION
to the order_groups claim above:
  Re-confirmed kalshi_ws.py still subscribes only to ticker + orderbook_delta
  (kalshi_ws.py:329) and kalshi_client.py has no order_group/fill support at
  all -- that part of the problem statement still holds.
  But fetched docs.kalshi.com/api-reference/order-groups/create-order-group
  directly and Kalshi's order_groups is NOT an OCO mechanism -- the original
  "linked orders where one fill auto-cancels the others" claim is wrong.
  It's actually a risk-management circuit breaker: contracts_limit sets a
  max matched-contracts count over a rolling 15-SECOND window; once hit,
  ALL orders in the group are cancelled and no new ones can be placed until
  reset. There is no field or documented behavior for "one order filling
  cancels the others" -- that's a self-built OCO concept this bot would
  still have to implement client-side (rest both legs, watch for either to
  fill via polling or the `fill` WS channel, cancel the other manually).
  order_groups' real use case is closer to protecting a market maker's
  resting quotes from a fast adverse-selection burst, not linking a
  take-profit and a stop-loss together -- still possibly useful for THIS
  bot's resting exits (cap how much can fill against a stale resting order
  in a 15s window) but as an extra safety layer, not as the OCO primitive
  the fix plan below assumed it would be.
  The WebSocket `fill` channel IS real and confirmed subscribable via
  `channels: ["fill"]` (docs.kalshi.com/websockets/websocket-connection) --
  that half of the plan holds.
  The market-maker rebate claim also needs correcting: Kalshi runs several
  DIFFERENT incentive programs, not one. Fetched
  help.kalshi.com/en/articles/13823851-liquidity-incentive-program directly:
  it pays a $10-$1,000/day share of a reward pool (Score / Total Participants'
  Scores * Reward Pool) for maintaining resting-order "Target Size" (100-
  20,000 contracts), NOT a flat percentage-of-fees rebate -- doesn't match
  the original "1%, capped $7,000/week" figure at all. Kalshi also has a
  separate Volume Incentive Program, Liquidity Provider Program, Combo
  Incentive Program, and a formal fee-rebate Market Maker Program referenced
  in CFTC filings (which may be where the original 1%/$7,000 figure came
  from, unconfirmed) -- these are genuinely different programs with
  different mechanics and eligibility, not one program with one number.
  Any real revenue-angle evaluation needs picking a specific program and
  reading its actual terms once this bot is resting real exit orders, not
  citing a single rebate percentage upfront.

What the fix looks like:
  When a live position opens, immediately rest a GTC sell at the take-profit
  price; implement OCO linking CLIENT-SIDE (order_groups does not do this
  for you -- see correction above): watch for either the take-profit or the
  stop leg to fill (via the `fill` WS channel or polling) and cancel the
  other manually. Subscribe to the WS fill channel to replace/(augment)
  _poll_pending_orders polling. Exits then execute continuously regardless
  of scan cadence, and resting exits get maker pricing instead of taker.
  order_groups' contracts_limit could still be layered in as an extra
  safety cap on a resting exit order's fill rate, separately from the OCO
  logic above.

  Note (revenue-angle, needs its own research pass once this ships -- see
  the multiple-distinct-programs correction above): don't assume a single
  rebate number; identify which Kalshi incentive program actually applies
  to this bot's real resting-order pattern and read its current terms

  Also see the endpoint-deprecation entry at the top of this file: Kalshi's
  order-mutation endpoints (used here for placing/canceling the resting
  exit orders) were migrated to /portfolio/events/orders this session (see

[RESOLVED 2026-07-12] MAKER EXECUTION UPGRADES — REPRICING + MAKER-AWARE
  FEE MODEL

PARTIALLY RESOLVED 2026-07-12: problem 2/3 (maker-aware fee model) is done;
  problems 1/4 (repricing/chase logic, fill-quality instrumentation) are

  and exactly $0 here, confirming utils.py's original "Maker orders pay 0%"

  (0.07 x p x (1-p) vs flat-fraction-of-winnings) turned out to be moot in
