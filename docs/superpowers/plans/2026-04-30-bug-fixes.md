## Informational answers (no code needed)

### Should big wins be weighted more than small losses in Brier?
Brier score (`(p - outcome)²`) treats all predictions equally by design — a 0.9 prediction that loses costs the same as a 0.1 prediction that wins (both score 0.01). This is intentional: it rewards calibration, not P&L. If you want P&L-weighted scoring, that's Kelly/expected-value logic, which the system already does for sizing. **No change needed here.**

### Win rate and Brier score are bad — what to do?
With fewer than ~200 settled trades, scores are noisy. Things that meaningfully help: (a) run backtest with `--days 180` to see if the model was better historically, (b) check if `learned_weights.json` is stale (`cmd_walkforward` → save weights), (c) look at the "worst calls" table in backtest output and see if a particular city or condition type is pulling scores down, (d) run calibration (`py main.py calibrate`). **No code change — addressed in Task 9 via in-app guidance.**

### Stop loss — is it working correctly?
Yes. `paper.py::check_stop_losses()` correctly exits when unrealized loss > `cost / STOP_LOSS_MULT`. Default `STOP_LOSS_MULT=2` means exit at 50% loss. The cron fires it before early-exit signals — that ordering is intentional (hard stop takes priority). **No change needed.**

### Monte Carlo clamping — is it a problem?
No. `monte_carlo.py` clamps `win_prob` to `[0.05, 0.90]` to guard against stale/bad data producing unrealistic simulations. A 95% win probability claim on a weather market should be treated with suspicion. The clamp slightly deflates extreme expected values — this is conservative and correct. **Task 10 adds a UI note so users know why.**
