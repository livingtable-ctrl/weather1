## LAYER 1 — EXECUTION CORRECTNESS

### L1-A | CRITICAL | Buy-NO API semantics ✅ VERIFIED-CORRECT
**File:** `kalshi_client.py` L247–278, `main.py` L2879–2886
**Finding:** `side="no" action="buy"` IS correct. Kalshi's REST API treats `side` and
`action` as orthogonal independent fields. Buying NO contracts requires `side="no" action="buy"`
with `no_price_dollars`. Using `side="yes" action="sell"` would close an existing YES position,
not open a new NO position.
**Code is correct** — `kalshi_client.py` lines 274–277 correctly routes `no_price_dollars`
when `side == "no"`. All call sites pass `side` through from the recommendation unchanged.
**Regression tests added:** `tests/test_kalshi_client.py::TestPlaceOrderApiSemantics` (3 tests)
— verifies body contains `no_price_dollars` for NO orders and `_place_live_order` calls
`place_order(side='no', action='buy')` not `place_order(side='yes', action='sell')`.

### L1-B | HIGH | Stale price at order time
**File:** `main.py` L2693–2700, `cron.py` L518–693
**Check:** `market_prob` in the analysis dict was computed during the batch scan.
By the time `_auto_place_trades` executes, the market may have moved.
The bot does NOT re-fetch live price before placing.
**Invariant:** Entry price used for sizing must reflect the market at order time, not scan time.

### L1-D | HIGH | Silent paper order failure
**File:** `main.py` L2853–2859
**Check:** Failed `place_paper_order` calls are caught and logged at WARNING only.
No user-visible alert. Signal is consumed but trade is not recorded.
**Invariant:** Every placement failure must be clearly surfaced to the operator.

## LAYER 2 — PRICE AND MATH BUGS

### L2-A | CRITICAL | NO-side entry price
**File:** `weather_markets.py` L3825
**Code:** `entry_price = prices["yes_ask"] if rec_side == "yes" else prices["no_bid"]`
**Check:** `no_bid` = what you receive selling NO. `no_ask` = what you pay buying NO.
For a market with YES bid/ask = 38¢/42¢: NO bid/ask = 58¢/62¢.
Using `no_bid=58¢` understates entry cost by 4¢. Kelly is inflated, P&L is wrong.
**Invariant:** `entry_price` for BUY NO = `1 - yes_bid` (the NO ask).

### L2-B | CRITICAL | Fee-free Kelly
**File:** `weather_markets.py` L3828
**Code:** `kelly = kelly_fraction(blended_prob, entry_price)` — no `fee_rate` arg
**Check:** The fee-adjusted path exists at L3908 and L3924. But the raw `kelly` key in
the result dict (L3986) is fee-free and used as sort/priority fallback in `_opp_sort_key`.
**Invariant:** Every sizing path must use `fee_rate=KALSHI_FEE_RATE`. Raw `kelly` must
never be used for actual position sizing.

### L2-C | CRITICAL | MOS hardcoded 50% weight
**File:** `weather_markets.py` L3682–3684
**Code:** `blended_prob = 0.5 * blended_prob + 0.5 * _mos_p_pre`
**Check:** MOS gets 50% unconditionally. If ensemble+NWS+clim agreed at 0.70 and MOS
says 0.35, result = 0.525 — direction flips from YES to NO. `blend_sources["mos"] = 0.5`
is added without halving existing weights → sum exceeds 1.0.
**Invariant:** MOS weight must come from the calibrated blend system. `sum(blend_sources.values()) == 1.0` after every injection.

### L2-D | HIGH | Cents/dollar conversion
**File:** `weather_markets.py` L1653–1658
**Check:** `v > 1` threshold. Value of exactly 1 (1¢) is not divided by 100.
`or 0` chaining at L1648 bypasses valid `0` values.
**Invariant:** All prices after `to_float()` must be in `[0.0, 1.0]`.

### L2-E | HIGH | Edge gate uses net_edge not adjusted_edge
**File:** `cron.py` L576, L615–618
**Check:** Gate uses `net_edge`. `adjusted_edge = net_edge * edge_confidence_factor`.
For a 5-day-out market, `edge_confidence` ≈ 0.5. A trade with `net_edge=0.20` but
`adjusted_edge=0.10` qualifies as STRONG and is auto-placed.
**Invariant:** Gate threshold should apply to `adjusted_edge`.

## LAYER 3 — POSITION MANAGEMENT

### L3-A | CRITICAL | No portfolio Kelly cap
**File:** `paper.py` L966–968, `main.py` L2683
**Check:** Global cap halts new trades when dollar exposure ≥ 50% of starting balance.
But each trade's Kelly is computed independently. 10 positions at Kelly=10% each = 100%
total Kelly exposure with no portfolio-level check.
**Invariant:** Sum of Kelly fractions across all open positions must be bounded.

### L3-B | HIGH | Exit recommendations never auto-executed
**File:** `paper.py` L1266–1319, `main.py` L2285–2341
**Check:** `check_model_exits` returns recommendations but does NOT call `close_paper_early`.
The caller in `cmd_watch` only prints to console. Only `_check_early_exits` actually closes.
**Invariant:** Exit recommendations must be acted upon, not just logged.

### L3-C | HIGH | Position dedup across restarts
**File:** `main.py` L2614–2615
**Check:** `open_tickers` rebuilt fresh each call from `get_open_trades()`. Cross-run dedup
relies solely on `execution_log.was_traded_today`. If this uses in-memory state, a restart
allows re-entering the same market.
**Invariant:** Dedup must survive process restarts.

## LAYER 4 — LEARNING SYSTEM INTEGRITY

### L4-A | CRITICAL | ABTest always returns value=None
**File:** `ab_test.py` L186–188
**Code:** `return chosen, None` — value is always None
**Check:** If callers use the returned value as the variant parameter (e.g., min_edge
threshold), they get `None` and fall back to a default — potentially the same variant
every time, making A/B testing inert.
**Invariant:** `get_active_variant` must return the actual variant value, not None.

### L4-B | HIGH | Null city pollutes bias
**File:** `main.py` L2835–2846, `tracker.py` L607–616
**Check:** If `enrich_with_forecast` fails to parse city, `log_prediction(city=None)`.
In `get_quintile_bias`, `city=None` skips the city filter — bias computed across all cities.
**Invariant:** `log_prediction` with `city=None` must be excluded from bias queries.

### L4-C | HIGH | Bias correction with 5 samples
**File:** `tracker.py` L627, L647–648
**Check:** `min_samples=5`. With 5 samples per quintile per city per month, one outlier
trade shifts `blended_prob` by ±20%. No shrinkage toward 0 for small samples.
**Invariant:** Bias correction should shrink toward 0 as sample count decreases.

### L4-D | MEDIUM | Stale learned weights
**File:** `weather_markets.py` L1419–1435
**Check:** `load_learned_weights()` reads `data/learned_weights.json`. No TTL check.
If last backtest was months ago, weights are stale.
**Invariant:** `learned_weights.json` must have `generated_at` timestamp checked at load.

## LAYER 5 — DATA PIPELINE

### L5-A | CRITICAL | Cache TTL misaligned with NWS update cycle
**File:** `forecast_cache.py` L16, `weather_markets.py` L238–240, L342–372
**Check:** 4h flat TTL. NWS updates at ~02, 08, 14, 20 UTC. A cache entry written at
07:59 UTC expires at 11:59 — missing 4h of improved model data. `_ttl_until_next_cycle`
exists but is not wired into cache expiry.
**Invariant:** Cache TTL must align to next NWS model cycle, not a flat 4h.

### L5-B | CRITICAL | City detection substring match
**File:** `weather_markets.py` L1845–1846
**Code:** `elif "LA" in ticker_up or "los angeles" in title:`
**Check:** Any ticker containing "LA" anywhere matches Los Angeles. A ticker like
`KXHIGHFLAT` would be misclassified. Wrong city = wrong forecast coordinates.
**Invariant:** City detection must use anchored series prefix matching, not bare substring.

### L5-C | HIGH | Circuit breaker 6h silence
**File:** `weather_markets.py` L41–48
**Check:** `failure_threshold=6` for forecast CB, but 3 parallel model fetches can record
3 failures simultaneously. After 2 such bursts, circuit opens for 6h. All analysis silently
returns None for an entire trading session.
**Invariant:** Circuit breaker recovery timeout must be proportional to actual API MTTR.

### L5-D | HIGH | Chicago station mismatch
**File:** `weather_markets.py` L103–123, `metar.py` L163
**Check:** `CITY_COORDS["Chicago"] = (41.9803, -87.9090)` — near O'Hare (KORD).
`MARKET_STATION_MAP["Chicago"] = "KMDW"` — Midway. O'Hare and Midway can differ 2–5°F.
**Invariant:** `CITY_COORDS`, `_metar_station_for_city`, and Kalshi's settlement station must all agree per city.

### L5-E | MEDIUM | date.today() is local time
**File:** `weather_markets.py` L3206, L1941
**Check:** `date.today()` uses local system timezone. On a UTC server, fine. On UTC-5
(Eastern), at 9 PM Eastern on Dec 31: `date.today()=Dec 31` but UTC=Jan 1. A Jan 1
market has `days_out=1` in Eastern, `days_out=0` in UTC — triggers different code paths.
**Invariant:** All `days_out` calculations must use `datetime.now(UTC).date()`.

## LAYER 6 — PROBABILITY CALIBRATION

### L6-A | CRITICAL | Backtest ensemble centered on actual outcome
**File:** `backtest.py` L100–104
**Code:** `result = [exact + random.gauss(0, sigma) for _ in range(50)]`
**Check:** `exact` is the historical observation (the actual outcome). Ensemble is centered
on the true answer by construction. The live model is centered on the model's forecast
which may be 5°F off. Brier score looks excellent in backtest but doesn't reflect
live model accuracy.
**Invariant:** Backtest ensemble must use model forecast at prediction time, not actual outcome.

### L6-B | HIGH | Gaussian double-counted in blend
**File:** `weather_markets.py` L3433–3436
**Code:** `ens_prob = 0.70 * ens_prob + 0.30 * gaussian_blend`
**Check:** `ens_prob` is modified in-place then fed into the final weighted blend labeled
as "ensemble". The Gaussian is counted once here and once implicitly through the blend weight.
**Invariant:** Gaussian contribution must be a separate named source, not overwrite `ens_prob`.

### L6-C | HIGH | "between" markets skip Gaussian
**File:** `weather_markets.py` L3383–3392
**Check:** `p_win_gaussian = None` for "between" condition types. These markets use only
ensemble member counting. With <10 members, probability is very noisy (0, 0.1, 0.2... steps).
**Invariant:** "between" markets need smoothed probability estimate, not raw member fraction.

### L6-D | HIGH | METAR lock-in hardcoded at 0.90
**File:** `metar.py` L23
**Code:** `_LOCK_IN_CONFIDENCE = 0.90`
**Check:** 3°F margin at 2 PM → same 90% confidence as 15°F margin at 10 PM. Kelly on
a 0.90 probability trade is very high. Near-threshold afternoon lock-ins are over-bet.
**Invariant:** Lock-in confidence must scale with temperature clearance and time of day.

### L6-E | MEDIUM | blend_sources sum > 1.0
**File:** `weather_markets.py` L3627–3684
**Check:** After main blend, `sum(blend_sources.values()) == 1.0`. After MOS injection,
`blend_sources["mos"] = 0.5` without halving existing weights → sum = 1.5.
**Invariant:** `sum(blend_sources.values()) == 1.0` after every blend step.

## LAYER 7 — PAPER P&L INTEGRITY (added)

### L7-A | CRITICAL | Paper trade settlement accuracy
**File:** `paper.py` — `settle_paper_trade()`
**Check:** How does the bot determine if a paper trade won or lost? Does it call the Kalshi
API for the settlement result and parse the correct field? If YES settling at $1.00 is parsed
as a loss (or vice versa), the measured win rate is inverted.
**Invariant:** A YES trade wins if and only if the market settles YES. Verify the settlement
price field and comparison logic.

### L7-B | HIGH | Paper fill price: mid vs ask
**File:** `paper.py` — `place_paper_order()`
**Check:** Does paper mode record the fill at mid-price, bid, or ask? If filled at mid but
real fills would be at ask, every paper trade appears half-spread more profitable than live.
Paper P&L is systematically optimistic.
**Invariant:** Paper fill price must match the price actually used: `yes_ask` for YES, `no_ask` for NO.

### L7-C | HIGH | Edge overstated by half-spread
**File:** `weather_markets.py` L3865, `main.py` L2402
**Check:** `edge = blended_prob - market_prob` where `market_prob = implied_prob = mid`.
But entry is at `yes_ask` (= mid + half_spread). Real edge at entry =
`blended_prob - yes_ask` which is smaller by half-spread (~2–5%).
A 7% apparent edge may be only 4% real edge after spread.
**Invariant:** Edge calculation must use ask price, not mid, as the market benchmark.

### L7-D | HIGH | time_decay_edge direction
**File:** `weather_markets.py` L2642–2668
**Check:** Does `time_decay_edge()` reduce edge as close time approaches (correct — less
time for the model to be right), or does it increase it (wrong — would inflate Kelly
on near-expiry markets)? Also check: does it return a multiplier or an additive adjustment?
**Invariant:** Edge must decrease (or stay flat) as `hours_to_close` decreases.

## LAYER 8 — PRECIPITATION / SIZING / SIGMA

### L8-A | CRITICAL | Precip and snow fast paths unaudited
**File:** `weather_markets.py` L2738–3029 (`_analyze_precip_trade`), L2903–3029 (`_analyze_snow_trade`)
**Check:** These functions are called at L3152–3167 and return before any temperature
logic runs. They have their own probability pipeline, own Kelly calculation, own blend
logic. None of the other audit items apply to them. Audit each for:
- Correct probability direction (does P(precip > threshold) use the right CDF direction?)
- Fee-aware Kelly (same L2-B risk applies independently)
- Entry price using ask not bid or mid
- Edge gate applied before trade is recommended
**Invariant:** All invariants from L2-A, L2-B, L2-C, L7-C must hold in the precip/snow
paths independently of the temperature path.

### L8-B | HIGH | kelly_quantity truncation to zero
**File:** `paper.py` — `kelly_quantity()` and `place_paper_order()`
**Check:** `int(dollars / price)` truncates fractional contracts to 0.
For a $5 Kelly allocation on a 60¢ contract: `int(5 / 0.60) = 8`. Fine.
For a $0.45 Kelly allocation on a 60¢ contract: `int(0.45 / 0.60) = 0`. Trade silently skipped.
With low bankroll or small Kelly fractions, many trades may be producing 0 contracts.
Also check: is there a minimum order size (e.g., 1 contract = 60¢) that the bot enforces?
If not, a 0-contract order may be submitted to the API and fail silently.
**Invariant:** `kelly_quantity()` must return at least 1 when `dollars >= price`, and
must clearly log/skip when allocation is below minimum.

### L8-C | HIGH | Historical sigma ≠ forecast RMSE
**File:** `weather_markets.py` — `get_historical_sigma(city, month)`
**Check:** This function returns the standard deviation of observed temperatures over
historical data — climatological variability. For NYC in January this might be 8–12°F.
But forecast RMSE for a 1-day-ahead GFS forecast is typically 2–4°F.
Using climatological sigma makes `_forecast_probability()` and `gaussian_probability()`
treat forecasts as far more uncertain than they are. A forecast of 72°F for a 70°F
threshold with sigma=10°F gives P(above 70) ≈ 0.58. With correct sigma=3°F: P ≈ 0.75.
The model systematically underestimates its own certainty, producing edges that are
too small and causing the bot to undersize or skip genuinely profitable trades.
**Invariant:** Sigma used in probability calculations must be forecast RMSE (model
error vs observed), not climatological variability (observed day-to-day spread).
